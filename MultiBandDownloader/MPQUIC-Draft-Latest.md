## 概要 (Abstract)

本文書は、単一の QUIC コネクションにおいて複数のパスを同時に利用可能とするための、QUIC プロトコルのマルチパス拡張を規定する。複数パスの作成・削除・管理を行うための明示的なパス識別子を導入する。本文書はアドレスの発見や管理、および QUIC を使用するアプリケーションが複数パスにわたってトラフィックをスケジュールする方法は規定しない。

## 1. はじめに (Introduction)

本文書は、QUIC バージョン 1 [QUIC-TRANSPORT] の拡張を規定し、同一または異なる 4-tuple（送信元/宛先ポート番号および送信元/宛先 IP アドレスからなる）を用いて、単一のコネクション上で複数のパスを同時に利用できるようにする。

[QUIC-TRANSPORT] のセクション 9 で規定されるコネクションマイグレーションは、ピアに対し、新しい優先パス経由での送信に切り替えるよう指示し、成功した場合は古いパスに関連するリソースを解放する。本文書で規定するマルチパス拡張は、このメカニズムを基盤としつつ、パス識別子（path ID）を導入してコネクション ID とパケット番号空間をパスごとに管理し、複数パスの同時利用を可能にする。したがって、本拡張におけるパスは path ID によって定義される。このため、同一の 4-tuple に対して複数のパス／path ID を持つことが可能であることに注意すること。

パケットのコネクション ID は、当該パケットを path ID（およびそれに対応するパケット番号空間）にバインドする。すなわち、各コネクション ID はちょうど 1 つの path ID に関連付けられるが、通常は各 path ID に対して複数のコネクション ID が発行される。同じ path ID が両方向で使用され、初期パスは 0 から始まる。Path ID は単調増加で生成され、再利用できない。

この拡張は複数のパケット番号空間を使用し、各パスに 1 つずつ割り当てる。各 path ID 固有のパケット番号空間はパケット番号 0 から始まる。これにより、各パスは [QUIC-TRANSPORT] のように、送信および受信パケットに対して個別のパケット番号状態を維持する。複数のパケット番号空間を使用することで、[QUIC-RECOVERY] で定義される損失検出および輻輳制御メカニズムを、パスごとに直接的に適用できる。ただし、複数パケット番号空間の使用には、パスとそのパケット番号空間を識別するために非ゼロのコネクション ID が必要であり、また path ID を含む変更された AEAD 計算（セクション 2.4 参照）が必要となる。

このため、本拡張は [QUIC-TRANSPORT] のセクション 9 におけるパス管理仕様からの逸脱を規定するものであり、本拡張のサポートを示すための新たなトランスポートパラメータが必要となる（セクション 2.1 参照）。

さらに本文書は、必要となるパス管理メカニズムとして、パス開始（セクション 3.1）、パスごとのコネクション ID の取り扱い（セクション 3.2）、優先パス使用のシグナリング（セクション 3.3）、放棄されたパスの明示的な削除（セクション 3.4）を規定する。本拡張において、QUIC サーバはパスの作成を開始することはできないが、クライアントによって作成された新しいパスを検証する必要があることに注意すること。

本拡張ではアドレスの発見および管理は対象外とする。本文書で規定する拡張は、アドレス管理とパスの設定／破棄の実際の判断がアプリケーションによって処理される場合、そのまま使用できる。本文書は将来の拡張が残るシナリオに対処するメカニズムを定義することを妨げない。

さらに、本文書では、同時に開かれた複数パスを使ってパケットを送る方法を定義するスケジューリングアルゴリズムは規定しない。これらはアプリケーションの要件によって異なるため、セクション 5 で基本的な実装ガイダンスのみを論じる。本拡張は様々なスケジューリングアルゴリズムと共に使用でき、たとえばフェイルオーバーのサポートから、開いた全パスにわたる集約容量の同時使用に至るまで対応可能である。複数パスを同時（並列）に利用するためのスケジューリングアルゴリズムを定義する IETF 仕様は、現時点では存在しない。

具体的に、Wi-Fi とモバイルネットワーク間のフェイルオーバーはマルチパスのよく知られたユースケースであるが、これは伝送の中断を避けるために 2 つのパスを一時的に同時利用するに過ぎない。一方、パスの同時利用は一般に、本文書で規定する以上の検討を必要とする。たとえば、ストリームデータが異なる遅延を持つ複数のパスに分配される場合、性能への悪影響を避けるための考慮が必要となる。

QUIC の運用上の考慮事項は [RFC9312] で扱われている。これらは本文書で定義する拡張を使用する QUIC コネクションにも適用される。追加の複雑さとして、アプリケーションが監視されるパスと監視されないパスの組み合わせを使用する可能性があるが、その複雑さは [QUIC-TRANSPORT] で定義されたパスマイグレーションの使用時にすでに存在する。

## 2. コネクションライフサイクルとパケット保護 (Connection Lifecycle and Packet Protection)

本文書は、マルチパス拡張のサポートを示すため、新たなトランスポートパラメータ `initial_max_path_id` を定義する。両エンドポイントのいずれかが `initial_max_path_id` トランスポートパラメータを通知しない場合、両エンドポイントは本文書で定義されるいかなるフレームやメカニズムも使用してはならない (MUST NOT)。両エンドポイントが `initial_max_path_id` トランスポートパラメータを通知する場合、ハンドシェイクが完了すると、セクション 2.4 で規定する新しい AEAD 利用がすべての 1-RTT パケットに適用され、セクション 3 で規定するように新しいパスを利用できる。セクション 4 で規定するように、本文書で定義される新しいフレームは 1-RTT パケットでのみ送信できる。

トランスポートパラメータのネゴシエーションにより、マルチパス拡張の実装の段階的な展開が可能となる。

### 2.1. initial_max_path_id トランスポートパラメータ

新しいトランスポートパラメータは以下のとおり定義される。

- **initial_max_path_id**（パラメータ ID `0x3e`）: パラメータ値は、コネクション開始時にエンドポイントが維持する意思のある最大の path ID を指定する可変長整数である。この値は、ノンス計算上の制約（セクション 2.4 参照）により、path ID の最大許容値である 2^32-1 を超えてはならない (MUST NOT)。

`initial_max_path_id` トランスポートパラメータは、コネクション中に使用できる開いたパスの初期最大数を制限する。たとえば、`initial_max_path_id` が 1 に設定されている場合、ピアによって発行されるべきコネクション ID は、path ID 0 と 1 に関連付けられたもののみである。エンドポイントが値 0 の `initial_max_path_id` トランスポートパラメータを受信した場合、ピアは追加パスを直ちに許可することなくマルチパス拡張を有効化することを意図している。

`initial_max_path_id` パラメータの設定は、同じ値の MAX_PATH_ID フレーム（セクション 4.6）を送信するのと等価である。後でより多くのパスの使用を許可するため、エンドポイントは MAX_PATH_ID フレームを送信して許容される最大 path ID を増加できる。

`initial_max_path_id` トランスポートパラメータの値が 2^32-1 より大きいものを受信した場合、受信側はタイプ TRANSPORT_PARAMETER_ERROR のエラーでコネクションをクローズしなければならない (MUST)。

`initial_max_path_id` トランスポートパラメータを通知する際、エンドポイントは長さが非ゼロのソースおよび宛先コネクション ID を使用しなければならない (MUST)。`initial_max_path_id` トランスポートパラメータを受信し、それを運ぶパケットが長さゼロのコネクション ID を含む場合、受信側はこれをタイプ PROTOCOL_VIOLATION のコネクションエラーとして扱い、コネクションをクローズしなければならない (MUST)。

ノンス長が 12 バイト未満の暗号スイートは、マルチパス拡張と共に使用できない。そのような暗号スイートが選択され、かつマルチパス拡張のサポートが有効である場合、エンドポイントはタイプ TRANSPORT_PARAMETER_ERROR のエラーでハンドシェイクを中止しなければならない (MUST)。

`initial_max_path_id` パラメータは、後続のコネクションで使用するために記憶してはならない (MUST NOT)（[QUIC-TRANSPORT] のセクション 7.4.1）。

### 2.2. 他のトランスポートパラメータとの関係

QUIC マルチパス拡張を使用する場合、`active_connection_id_limit` トランスポートパラメータ [QUIC-TRANSPORT] は、パスごとのアクティブなコネクション ID の最大数を制限する。[QUIC-TRANSPORT] のセクション 5.1.1 で定義されているとおり、発行されてリタイアされていないコネクション ID はアクティブとみなされる。

エンドポイントが `disable_active_migration` トランスポートパラメータを受信した場合、ピアのハンドシェイクアドレスへの新しいパスの確立は禁止される。ただし、他のピアアドレス（たとえばピアの `preferred_address` で運ばれるもの）への追加パスの確立は直ちに有効である。

サーバが `preferred_address` トランスポートパラメータを使用する場合、クライアントは初期サーバアドレスとこのパラメータに含まれるアドレスをマルチパス用に同時に使用できると仮定してはならない（[QUIC-TRANSPORT] のセクション 9.6.2）。同じローカルアドレスでの優先アドレスの使用は、path ID を変更しないマイグレーションイベントとみなされる。したがって、`preferred_address` トランスポートパラメータで指定されたコネクション ID の path ID は 0 となる。

### 2.3. 0-RTT および 1-RTT における ACK と PATH_ACK の扱い

PATH_ACK フレーム（セクション 4.1 参照）は 1-RTT パケットを確認応答するために使用される。[QUIC-TRANSPORT] のセクション 19.3 で規定される ACK フレームと比較すると、PATH_ACK フレームはパス固有のパケット番号空間を識別するため、追加で path ID を含む。マルチパス拡張で ACK フレームが使用される場合、それは path ID 0 のパスのパケットを確認応答する。マルチパスサポートはハンドシェイク中には不明であるため、Initial パケットおよび Handshake パケットの確認応答は ACK フレームを使用して送信される。

ハンドシェイクがマルチパス拡張のサポートで完結した後、エンドポイントは ACK フレームの代わりに PATH_ACK フレームを使用すべきである (SHOULD)。これには、path ID 0 を使用してまだ確認応答されていない 0-RTT パケットも含まれる。エンドポイントは、0-RTT パケットまたは 1-RTT パケットを確認応答する ACK フレームを引き続き処理しなければならない (MUST)。たとえば、送信者は後の使用のためにマルチパスサポートをネゴシエートし、しばらくの間は path ID 0 の初期パスのみを保持する場合がある。この単一パスの期間中、送信者は ACK フレームを送信することを好む可能性がある。

### 2.4. ハンドシェイク完了後のノンス計算

[QUIC-TLS] のセクション 5.3 では AEAD の使用、特にパケット保護 IV とパケット番号を組み合わせて形成されるノンス N の使用を規定している。複数のパケット番号空間が使用される場合、パケット番号だけではノンスの一意性を保証できない。したがって、マルチパス拡張が使用される場合、1-RTT パケット用のノンス N は、パケット保護 IV をパケット番号および path ID の 32 ビットと組み合わせて計算される。ノンスの一意性を保証するため、path ID はセクション 2.1 で規定するように最大値 2^32-1 に制限される。

ノンスを計算するため、96 ビットの「Path-and-Packet-Number (PPN)」は、ネットワークバイトオーダーの path ID 32 ビット、2 つのゼロビット、およびネットワークバイトオーダーの再構築された QUIC パケット番号 62 ビットで構成される（図 1 に図示）。

```
  PPN {
    Path ID (32),
    Zeroes (2) = 0b00,
    Packet Number (62)
  }

           図 1: 96 ビットの Path-And-Packet-Number
```

IV 長はノンス長と等しい。IV が 96 ビットより大きい場合、path-and-packet-number は IV のサイズまで左側にゼロでパディングされる。パディングされたパケット番号と IV の排他的論理和が AEAD ノンスを形成する。ノンス長が 12 バイト未満の AEAD アルゴリズムは、QUIC マルチパス拡張と共に使用できない。次の図は 96 ビット IV の場合を示す。

```
IV(12);
N(12) = IV xor PPN;

                図 2: ノンス計算
```

たとえば、IV 値が `0x6b26114b9cba2b63a9e8dd4f`、path ID が `3`、パケット番号が `54321`（16 進値 `0xd431`）の場合、ノンスは以下のように `0x6b2611489cba2b63a9e8097e` に設定される。

```
   IV:      6b26114b9cba2b63a9e8dd4f
⊕  PPN:     00000003000000000000d431
------------------------------------
   Nonce:   6b2611489cba2b63a9e8097e

              図 3: ノンス計算の例
```

### 2.5. キーフェーズの更新プロセス

Key Phase ビットの更新プロセスは [QUIC-TLS] のセクション 6 で規定されている。本仕様において、キー更新の一般原則は変更されない。[QUIC-TLS] に従い、Key Phase ビットはパケットを保護するために使用されるパケット保護鍵を示すために用いられる。Key Phase ビットは、後続のキー更新ごとに切り替えられる。

ネットワーク遅延のため、古い鍵で保護されたパケットが新しい鍵で保護されたパケットより遅れて到着する場合があるが、受信側は連続する 2 回のキー更新の間に十分な間隔があると仮定すれば、Key Phase ビットだけを頼りに対応するパケット保護鍵を判定できる ([QUIC-TLS] のセクション 6.5)。

本仕様が使用される場合、エンドポイントは、前回のキー更新の受信を確認する確認応答を受け取った後、新しいキー更新を開始する前に、すべてのパスにわたる最大の Probe Timeout (PTO)（[QUIC-RECOVERY] のセクション 6.2 参照）の少なくとも 3 倍の時間を待つべきである (SHOULD)。この間隔は [QUIC-TLS] とは異なり、[QUIC-TLS] では単一パスの PTO の 3 倍であった。

復号鍵が破棄された後に到着するパケットはドロップされるため、最大 PTO の 3 倍を選択するのはトレードオフである。長い遅延はパケット損失の確率を減らすが、古い鍵を長く保持することはプロトコルのセキュリティに悪影響を与える可能性がある。最大 PTO の 3 倍を使用することは、すべてのパスにわたるパケット損失を最小化し、性能への影響を制限することを目的としている。

[QUIC-TLS] のセクション 5.4 に従い、Key Phase ビットは保護されるため、複数のパケットを同時に Key Phase ビットを反転させて送信しても、観測者から見て異なるパス間の活動がリンク可能になることはない。

### 2.6. コネクションクロージャ (Connection Closure)

CONNECTION_CLOSE フレームとその処理は [QUIC-TRANSPORT] から変更されない。これらはいずれの開いたパス上でも送信できる。[QUIC-TRANSPORT] のセクション 10.2 では、closing および draining コネクション状態は「現在の PTO の少なくとも 3 倍の時間持続すべき (SHOULD)」と規定している。本仕様が使用される場合、これらの状態は代わりにすべてのパスにわたる最大 PTO の少なくとも 3 倍の時間持続すべきである (SHOULD)。

## 3. パス管理 (Path Management)

マルチパスサポートを示すハンドシェイクが完了した後、エンドポイントは複数のパスを使用し始めることができる。[QUIC-TRANSPORT] のセクション 9 に準じて、本拡張ではクライアントのみが新しいパスを開くことができる。両エンドポイントが、同じ path ID が両方向で使用されるため、少なくとも 1 つの未使用かつ共通の path ID に対して利用可能なコネクション ID を発行している場合、クライアントは新しいパスを開くことができる。

本文書は、パスの開始（セクション 3.1）、パスごとのコネクション ID の発行と退役（セクション 3.2）、パスのステータス管理（セクション 3.3）、およびパスのクローズ（セクション 3.4）を規定する。ただし、本文書はクライアントがいつパスを開始または閉じるかの判断、または開いた複数パスを送信にどのように使うかは規定しない。

パス管理のため、本拡張はセクション 4 で以下のフレームを規定する。

- PATH_ABANDON（セクション 4.2 参照）
- PATH_STATUS_BACKUP（セクション 4.3 参照）
- PATH_STATUS_AVAILABLE（セクション 4.3 参照）
- PATH_NEW_CONNECTION_ID（セクション 4.4 参照）
- PATH_RETIRE_CONNECTION_ID（セクション 4.5 参照）
- MAX_PATH_ID（セクション 4.6 参照）
- PATHS_BLOCKED（セクション 4.7 参照）
- PATH_CIDS_BLOCKED（セクション 4.7 参照）

### 3.1. パスの開始と検証 (Path Initiation and Validation)

新しいパスを開くため、エンドポイントは未使用の path ID に関連付けられた新しいコネクション ID を使用しなければならない (MUST)。エンドポイントは、同じパス上でパケットを送信する意図がある場合、エンドポイントが受信したパケットで使用された path ID と同じ path ID に関連付けられたコネクション ID を使用しなければならない (MUST)。

新しいパスを使用したいクライアントは、当該パスに使用される 4-tuple を以前に検証していない限り、[QUIC-TRANSPORT] のセクション 8.2 に記述されるとおりピアのアドレスを検証しなければならない (MUST)。

新しいパス上でクライアントからパケットを受信した後、サーバが新しいパスを使うと判断した場合、サーバは当該パスに使用される 4-tuple を以前に検証していない限り、データを送信する前に ([QUIC-TRANSPORT] のセクション 8.2 に記述されるとおり) ピアのアドレスを検証しなければならない (MUST)。クライアントのアドレスが検証されるまで、[QUIC-TRANSPORT] のセクション 8 のアンチアンプリフィケーション制限が適用される。

エンドポイントが PATH_RESPONSE（[QUIC-TRANSPORT] のセクション 19.18）を送信する場合、PATH_CHALLENGE フレーム（[QUIC-TRANSPORT] のセクション 19.17）を含むパケットで使用されたパスと同じパス上で、同じ path ID に関連付けられたコネクション ID を使用して送信しなければならない (MUST)。

サーバは、PATH_CHALLENGE フレームを含まない、まだ使用されていない path ID 用のパケットを受信する場合がある。そのようなパケットは、有効なコネクション ID により正しく復号できれば有効である。

各エンドポイントは、最小 QUIC パケット MTU 1200 バイトがパス上でサポートされていることも検証しなければならない (MUST)。これは、初期パス検証時、または増幅制限が初期に妨げる場合は別途後で行うことができる ([QUIC-TRANSPORT] のセクション 8.2.1 参照)。

新しいパス上でパケットを受信し、このパスを確立したくないエンドポイントは、セクション 3.4 で規定するように、別のパス上で PATH_ABANDON を送信してパスをクローズすると想定される。

このパス用のアクティブなコネクション ID を持たない、または直ちに新しいパスを構成する他のリソースを欠くエンドポイントは、十分なリソースが利用可能になるまで PATH_RESPONSE の送信を遅延させることができる。長い遅延はピアに PATH_CHALLENGE の繰り返しと最終的には PATH_ABANDON の送信を引き起こす可能性があり、その場合はセクション 3.4 で規定する手順が適用される。

PATH_ACK フレーム（セクション 4.1 参照）はどのパス上でも返すことができる。確認応答されるパケットと同じパス上で PATH_ACK を送信することが好まれる場合（さらなるガイダンスはセクション 5.4 参照）、パス検証中に PATH_ACK フレームを PATH_RESPONSE フレームとバンドルすることが有益となり得る。

検証が成功すれば、クライアントはパスを引き続き使用できる。検証が失敗した場合、クライアントはパスを使用してはならず (MUST NOT)、パス開始試行に関連付けられた状態を削除できる。使用された path ID はいずれにせよ消費されるため、エンドポイントはセクション 3.4 で規定するように明示的にパスをクローズしなければならない (MUST)。

#### 3.1.1. パス確立の例

以下の例では、両エンドポイントが少なくとも 2 の `initial_max_path_id` 値を示しているものと仮定する。これは、両エンドポイントが path ID 0、1、2 を使用できることを意味する。なお、path ID 0 はすでに初期パスに使用されている。

```
   Client                                                  Server

   (既存の path 0 上で path 1 用の新しい CID を提供)
   1-RTT[X]: DCID=S0, PATH_NEW_CONNECTION_ID[C1, Seq=0, PathID=1] -->
           <-- 1-RTT[Y]: DCID=C0,
                         PATH_NEW_CONNECTION_ID[S1, Seq=0, PathID=1],
                         PATH_ACK[PathID=0, PN=X]
           <-- 1-RTT[Y+1]: DCID=C0, PATH_NEW_CONNECTION_ID[S2, Seq=0,
                                                            PathID=2]
   ...
   (path ID 1 を使用して新しいパス上でパケットの送信を開始)
   1-RTT[0]: DCID=S1, PATH_CHALLENGE[X] -->
        <-- 1-RTT[0]: DCID=C1, PATH_RESPONSE[X], PATH_CHALLENGE[Y],
                                             PATH_ACK[PathID=1, PN=0]
   1-RTT[1]: DCID=S1, PATH_RESPONSE[Y],
            PATH_ACK[PathID=1, PN=0], ... -->

                図 4: 新しいパス確立の例
```

図 4 において、エンドポイントはまず PATH_NEW_CONNECTION_ID フレームで利用可能な新しいコネクション ID を交換する（さらにセクション 3.2 で説明される）。この例では、クライアントは 1 つのコネクション ID（C1, path ID 1）を提供し、サーバは 2 つのコネクション ID（S1, path ID 1 および S2, path ID 2）を提供する。

クライアントは、PATH_CHALLENGE フレームを含むパケットを当該パス上で送信して新しいパスを開く前に、両側で同じ未使用 path ID 用の未使用コネクション ID が利用可能かをチェックする必要がある。この例では、セクション 3.2 で推奨されるように、利用可能な未使用 path ID のうち最小である path ID 1 が使用される。それぞれの場合において、クライアントは PATH_CHALLENGE フレームを送信する際、新しいパスの宛先コネクション ID として S1 を選択する。サーバは同じ path ID に関連付けられたコネクション ID C1 を使用して、PATH_RESPONSE と PATH_ACK をバンドルして応答する。

#### 3.1.2. プロービングおよびマイグレーションとの関係

[QUIC-TRANSPORT] のセクション 9.1 では「プロービング (probing)」フレームと「非プロービング (non-probing)」フレームの概念を導入している。少なくとも 1 つの「非プロービング」フレームを含むパケットは「非プロービング」パケットである。[QUIC-TRANSPORT] のセクション 9.2 で規定されるマイグレーションは、新しい（検証済みの）パス上で、ただし古いパスと同じ path ID を使用して、非プロービングフレームを含むパケットを送信することによって開始される。マルチパス拡張がネゴシエートされた場合、新しい未使用の path ID を持つ新しいパス上での「プロービング」または「非プロービング」のいずれのパケットの受信も、既存のパスのパス状態に影響を及ぼさない。したがって、当該パスのアンチアンプリフィケーション制限（[QUIC-TRANSPORT] のセクション 21.1.1.1 参照）および輻輳制御制限が尊重される限り、いかなるフレームも新しい path ID を持つ新しいパス上でいつでも送信できる。

エンドポイントは、アクティブな path ID に関連付けられたコネクション ID を持つパケットを受信し、そのパケットの 4-tuple が当該 path ID で現在使用されている 4-tuple と一致しない場合がある。これは [QUIC-TRANSPORT] のセクション 9.3 で規定するパスマイグレーションとして扱わなければならず (MUST)、パスマイグレーション中に使用されるすべてのコネクション ID は、マイグレーション対象のパスの現在の path ID に関連付けられていなければならない (MUST)。

#### 3.1.3. アドレス検証トークン

[QUIC-TRANSPORT] のセクション 9.3 で規定されているように、サーバは新しいクライアントアドレスの検証成功後にクライアントへ新しいアドレス検証トークンを送信することが期待される。クライアントは複数のトークンを受信することになる。後続のコネクションでトークンを使用することを検討する際、クライアントは前のコネクションで取得した複数のトークンの中から「正しい」トークンを選ぶことが難しい場合がある。クライアントは、[QUIC-TRANSPORT] のセクション 8.1.3 で規定される戦略、すなわち最後に受信したトークンを選ぶ戦略にフォールバックする可能性が高い。クライアントが「間違った」選択をした場合の問題を避けるため、サーバは以前に検証されたいずれのアドレスも検証可能なトークンを発行すべきである (SHOULD)。より多くのアドレスを含めることにより、トークンが将来役立つ確率が高まるが、トークンが大きくなるという代償がある。トークン使用に関するさらなるガイダンスは [QUIC-TRANSPORT] のセクション 8.1.3 にある。

### 3.2. コネクション ID の取り扱い

マルチパス拡張が使用される場合、エンドポイントは少なくとも path ID が 0 以外のすべてのパスについて、関連付けられたシーケンス番号と共に、それぞれの path ID を示すために PATH_NEW_CONNECTION_ID フレームと PATH_RETIRE_CONNECTION_ID フレームを使用しなければならない（[QUIC-TRANSPORT] のセクション 5.1.1 参照）。各 path ID は、初期値 0 から始まる独自のコネクション ID シーケンス番号空間を持つ。

エンドポイントは、path ID 0 の初期パスについても PATH_NEW_CONNECTION_ID と PATH_RETIRE_CONNECTION_ID を使用すべきである (SHOULD)。ただし、NEW_CONNECTION_ID と RETIRE_CONNECTION_ID の使用は依然として有効であり、エンドポイントはこれらのフレームを path ID 0 に対応するものとして処理する必要がある。

#### 3.2.1. 新しいコネクション ID の発行

ピアが新しいパスを開けるようにするため、ピアおよびローカルの最大 path ID 制限の最小値まで、未使用の各 path ID に対して少なくとも 1 つのコネクション ID を能動的に発行することが推奨される (RECOMMENDED)。

何らかの理由でエンドポイントがすべての未使用 path ID に対してコネクション ID を発行したくない場合、path ID の発行に不連続を導入すべきではない (SHOULD NOT)。なぜなら、パス開始には両側で同じ path ID 用の利用可能なコネクション ID が必要だからである。たとえば、最大 path ID 制限が 2 で、エンドポイントが範囲 [1, 2] 内の 1 つの path ID のみにコネクション ID を提供したい場合は、path ID 1 を選択すべきである（path ID 2 ではなく）。

同様に、エンドポイントは path ID を連続的に消費すべきである (SHOULD)、つまりパス作成時に。ただし、エンドポイントは順序外配信またはパス検証失敗のため、path ID の順序通りの使用で新しいコネクション ID やパス開始試行を受信することは期待できない。

各エンドポイントは、ピアから受信したコネクション ID の集合をパスごとに維持し、当該パス上でパケットを送信する際にはそのいずれかを使用できる（[QUIC-TRANSPORT] のセクション 5.1 も参照）。通常、（意図しない）マイグレーションイベント（[QUIC-TRANSPORT] のセクション 9.5）のため、使用中のすべてのパスに少なくとも 1 つの追加コネクション ID を提供することが望まれる。

[QUIC-TRANSPORT] のセクション 5.1 でさらに規定されているように、コネクション ID は同じコネクション上で複数回発行できないため、関連付けられた path ID にかかわらず、コネクションのスコープにおいて一意である。

エンドポイントは、MAX_PATH_ID フレーム（セクション 4.6）の Maximum Path Identifier フィールドより大きい path ID、または MAX_PATH_ID フレームをまだ受信していない場合は `initial_max_path_id` トランスポートパラメータの値より大きい path ID で、新しいコネクション ID を発行してはならない (MUST NOT)。より大きい path ID を持つフレームの受信は、セクション 4 で規定するコネクションエラーである。

エンドポイントは、未使用 path ID がないためにパスを開けない場合、最大 path ID 制限を増やすために MAX_PATH_ID フレームを送信するか（送信側によって制限されている場合）、または現在の制限が新しいパスの作成を妨げたことをピアに通知するために PATHS_BLOCKED フレーム（セクション 4.7）を送信すべきである (SHOULD)。

#### 3.2.2. コネクション ID のローテーションと退役

[QUIC-TRANSPORT] のセクション 5.1.2 では、エンドポイントはコネクション中いつでも使用するコネクション ID を別の利用可能なものに変更できることを示している。本文書で規定する拡張では、エンドポイントは同じ path ID に関連付けられた別のコネクション ID にのみローテーションしなければならない (MUST)。別の path ID に関連付けられたコネクション ID の使用は、新しいパスを開く試みとみなされる。

エンドポイントは、使用されていないコネクション ID を退役することが想定され、サーバは [QUIC-TRANSPORT] のセクション 5.1.2 で規定するように代替を提供することが期待される。したがって、PATH_RETIRE_CONNECTION_ID フレームを受信した場合、エンドポイントは当該パスがまだ開かれているなら、PATH_NEW_CONNECTION_ID フレームを使用してそのパス用の新しいコネクション ID を提供すべきである (SHOULD)。

ピアがすべてのアクティブなパスに対して、古いコネクション ID の退役後に PATH_NEW_CONNECTION_ID を使用して少なくとも 1 つの未使用コネクション ID を提供することが期待されるが、エンドポイントはコネクション ID を変更したいが当該パス用の未使用コネクション ID が利用できない場合、PATH_CIDS_BLOCKED（セクション 4.7 参照）を送信してもよい (MAY)。さらに、エンドポイントは、Maximum Path Identifier 値がより多くのパスを許可するにもかかわらず、新しいパスを開きたくて未使用 path ID 用の利用可能なコネクション ID がない場合にも、PATH_CIDS_BLOCKED フレームを送信してもよい (MAY)。

コネクション ID の退役は、当該コネクション ID に対応する path ID やその他のパスリソースを退役させない。なぜなら、パケット番号空間は path ID に関連付けられているからである。

PATH_RETIRE_CONNECTION_ID フレームを送信したピアは、退役したコネクション ID が使用されていたパス上でデータを送信し続けることができるが、その際は同じ path ID 用の異なるコネクション ID を使用しなければならない。

### 3.3. パスステータス管理

エンドポイントは、これらのフレームによって表現される優先順位でパス上でパケットを送信すべきであることをピアに通知するため、PATH_STATUS_BACKUP および PATH_STATUS_AVAILABLE フレーム（セクション 4.3 参照）を送信できる。なお、エンドポイントはピアの通知に従わない場合があるが、これらのフレームは依然としてピアのパス使用優先順位の明確なシグナルである。

各ピアは他方のピアとは独立にパス使用の優先順位を示す。つまり、ピアは同じパスに対して異なる使用優先順位を持つ可能性がある。データ送信側の判断によっては、ピアによって「バックアップ」と示されたパスの使用や、ローカルで利用可能な一部のパスの不使用につながる場合がある。

PATH_STATUS_AVAILABLE はパスが「利用可能 (available)」であることを示す。すなわち、利用可能なパス間でトラフィックを分割するために自身のロジックを使用するようピアに提案する。

PATH_STATUS_BACKUP は、パスがバックアップとしてのみ使用されるべきであることを提案する。すなわち、別のパスが利用可能で使用可能な場合、当該パス上でトラフィックを送信しないことを提案する。確立されたすべてのパスがバックアップパスとして示されている場合、どのパスを使用すべきかについてのガイダンスは提供されない。

同様に、特定のパスについてパス使用優先順位を示すフレームを受信していない場合、ピアの優先順位は不明であり、送信側はパスを使用すべきかを自身のローカルロジックに基づいて判断する必要がある。

エンドポイントが「利用可能」とマークされたパスで問題を検出してバックアップパスを使い始めた場合、ピアが壊れたパスの使用を避けるよう、自身のパス状態シグナルを更新することが推奨される (RECOMMENDED)。パス断絶を検出したエンドポイントは、ピアが当該パスを使い続けることを避け、バックアップパスへのより速いスイッチオーバーを可能にするため、PATH_ABANDON フレーム（セクション 3.4 参照）を送信することによってパスを明示的にクローズすることもできる。エンドポイントが、接続が再確立される可能性があるためパスを直ちに閉じたくない場合、PING フレームを使用して接続性の変化を迅速に検出し、タイムリーにスイッチバックすることが可能である。

PATH_STATUS_AVAILABLE および PATH_STATUS_BACKUP フレームは、古い情報を検出して無視するため、共通のパスごとのシーケンス番号空間を共有する。これは、たとえば異なるパスを使用して送信された場合に順序外で到着する可能性があるため必要である。詳細はセクション 4.3 を参照。

### 3.4. パスのクローズ

コネクション中のいつでも、各エンドポイントは、たとえばローカル接続性やローカル優先順位の変更に伴い、パスを放棄する判断を下すことができる。パスを放棄したいエンドポイントは、PATH_ABANDON フレーム（セクション 4.2 参照）を送信して明示的にパスをクローズしなければならない (MUST)。これは、パスをクローズする判断が、アイドルタイムアウト（セクション 5.9 参照）やパケット損失などの暗黙的なシグナルから生じる場合、およびローカルリソースの管理など他の理由でも当てはまる。

PATH_ABANDON フレームを送信するエンドポイントは、PATH_ABANDON で示された path ID 用にピアから受信したすべてのコネクション ID を直ちに退役したものとして扱わなければならず (MUST)、その後は当該パス上でいかなるパケットも送信できなくなる。なお、パスを放棄するとコネクション ID の退役が発生するが、その逆は真ではない。すなわち、関連付けられたコネクション ID を退役することはパスの放棄を示さない（さらにセクション 3.2 を参照）。

PATH_ABANDON フレームは、クローズされる予定のパスだけでなく、いずれの開いたパス上でも送信できる。特に、放棄予定のパス上の接続性が壊れていると予想される場合、PATH_ABANDON フレームは別の開いたパス上で送信することが推奨される (RECOMMENDED)。

エンドポイントが PATH_ABANDON フレームを受信した場合、まだ送信していなければ対応する PATH_ABANDON フレームを送信しなければならず (MUST)、また当該パスについてピアから受信したすべてのコネクション ID をそれぞれ直ちに退役したものとして扱わなければならない。これは、ピアから受信した退役したコネクション ID は送信に使用できなくなるが、ピアからのパケットは依然として転送中である可能性があることを意味する。したがって、ピアに発行したコネクション ID およびパスに関連付けられた番号空間の状態の知識は、PATH_ABANDON フレームを受信してから 3 PTO の間保持すべきである (SHOULD)。これは、セクション 3.4.2 で論じる偽のステートレスリセットパケットの生成を回避し、ピアからの順序が乱れた可能性のある残存パケットを確認応答するのに役立つ（セクション 3.4.3 参照）。

エンドポイントが、パス上でいかなるトラフィックも受信または送信する前に PATH_ABANDON フレームを受信することもあり得る。たとえば、クライアントがパスの開始を試み、パスが確立できない場合、PATH_ABANDON フレームを送信する（セクション 3.1 参照）。エンドポイントは、たとえば使用中の path ID シーケンスから穴を取り除くためなど、何らかの理由で未使用パスを放棄する判断を下す場合もある。これはエラーではない。

ピアが PATH_ABANDON フレームを送信したにもかかわらず、対応する PATH_ABANDON フレームを受信しない場合、パス状態を削除できない可能性がある。この予期しない動作の処理方法は実装に委ねられており、相互運用性には影響しない。エンドポイントが放棄されたパス用に発行されたコネクション ID を処理する意思がもうない場合、コネクションをクローズしてもよい (MAY)。ただし、PATH_ABANDON フレームの送信後少なくとも 3 PTO 待つべきである (SHOULD)。

パスが放棄された後、path ID は新しいパスに再利用してはならない (MUST NOT)。なぜなら、path ID はノンス計算の一部だからである（セクション 2.4）。

QUIC コネクションの唯一の開いたパスに対して PATH_ABANDON フレームが受信された場合、受信側ピアは CONNECTION_CLOSE フレームを送信し、closing 状態に入るべきである (SHOULD)。あるいは、クライアントは利用可能であれば新しいパスを開くことを試みてもよく (MAY)、パス検証が失敗した場合またはサーバから CONNECTION_CLOSE フレームを受信した場合にのみコネクションクロージャを開始する。同様に、サーバは CONNECTION_CLOSE フレームを送信する前に、新しいパス上でパケットが受信されるかを確認するため 1 PTO などの短い限定された時間待ってもよい (MAY)。

なお、[QUIC-TRANSPORT] の他の明示的なクロージングメカニズムは依然としてコネクション全体に適用される。特に、CONNECTION_CLOSE（[QUIC-TRANSPORT] のセクション 10.2）または Stateless Reset（[QUIC-TRANSPORT] のセクション 10.3）の受信はコネクションをクローズする。

#### 3.4.1. パスクローズの例

以下の例では、クライアントが path ID 0 のパスをクローズしたい。クライアントは、コネクション ID S1 を使用して path ID 1 のパス上で path ID 0 のパスを終了するため、PATH_ABANDON フレームを送信する。path ID 0 用の PATH_ABANDON フレームを受信した後、サーバも同じパス上でコネクション ID C1 を使用して、PATH_ACK フレームと共に path ID 0 用の PATH_ABANDON フレームを送信する。

```
Client                                                      Server

(クライアントは path ID 0 のパスを放棄するようサーバに伝える)
1-RTT[X]: DCID=S1 PATH_ABANDON[path ID=0]->
                          (サーバはクライアントにパスを放棄するよう伝える)
                   <-1-RTT[Y]: DCID=C1 PATH_ABANDON[path ID=0],
                                          PATH_ACK[PATH ID=1, PN=X]
1-RTT[U]: DCID=S1 PATH_ACK[path ID=1, PN=Y] ->

                図 5: パスをクローズする例
```

なお、PATH_ABANDON フレームが代わりに放棄予定のパス上で送信される場合、PATH_ABANDON フレームの後は放棄されたパス上でパケットを送信できないため、最後の確認応答も依然として別のパス上で送信する必要がある。

#### 3.4.2. 偽のステートレスリセットの回避

ネットワーク遅延のため、放棄されたパス上で送信されたパケットは、コネクション ID が退役された後、十分長い時間後に到着する可能性がある。ローカルコネクションにバウンドされていると認識されない場合、そのようなパケットは、ピアによる Stateless Reset パケットの送信をトリガーする。上述のセクション 3.4 で規定するように、PATH_ABANDON フレームを受信してから 3 PTO の間コネクション ID とパケット番号空間に関する知識を保持するという要件は、そのような偽のステートレスパケットの送信のリスクを減らすことを目的としているが、そのリスクを完全に回避することはできない。

[QUIC-TRANSPORT] のセクション 10.3 では、退役されたコネクション ID に関連付けられた Stateless Reset Token は Stateless Reset パケットを識別するために使用できないと規定している。放棄されたパス用にピアから受信したコネクション ID の即時退役は、ピアによって送信される偽の Stateless Reset パケットが QUIC コネクションのクローズを引き起こさないことを保証する。

#### 3.4.3. 放棄されたパスの PATH_ACK の取り扱い

エンドポイントが PATH_ABANDON フレームを送信した場合、ピアからの一部のパケットがまだ転送中である可能性がある。さらに、エンドポイントが PATH_ABANDON フレームを受信した場合、放棄されたパス上で順序が乱れたパケットを受信する可能性がある。エンドポイントは、パス状態がまだ保持されてそのようにできる場合、放棄されたパス上で受信したすべての未確認応答パケットに対して PATH_ACK フレームを速やかに送信すべきである (SHOULD)。

PATH_ACK フレームは、PATH_ABANDON フレームを送信した後はコネクション ID が直ちに退役されるため、放棄予定のパスとは異なるパス上で送信する必要がある。

エンドポイントが最終的にパスに関連付けられたすべての状態を削除した場合、パス上で送信されまだ確認応答されていないパケットは失われたとみなさなければならない (MUST)。放棄された path ID で受信した PATH_ACK フレームは、セクション 4 で規定するように静かに無視される。

## 4. 新しいフレーム (New Frames)

本文書で定義されるすべてのフレームは、1-RTT パケットでのみ送信されなければならない (MUST)。エンドポイントが異なるパケットタイプでマルチパス固有のフレームを受信した場合、タイプ PROTOCOL_VIOLATION のエラーでコネクションをクローズしなければならない (MUST)。

MAX_PATH_ID フレーム、または MAX_PATH_ID フレームをまだ受信していない場合は `initial_max_path_id` トランスポートパラメータで通知された Maximum Paths 値より大きい path ID を使用するマルチパス固有のフレームの受信は、タイプ PROTOCOL_VIOLATION のコネクションエラーとして扱わなければならない (MUST)。

エンドポイントがもう処理できない path ID（たとえばパスが放棄されている可能性があるため）を持つマルチパス固有のフレームを受信した場合、当該フレームを静かに無視しなければならない (MUST)。

### 4.1. PATH_ACK フレーム

PATH_ACK フレーム（タイプ `0x3e` および `0x3f`）は、[QUIC-TRANSPORT] のセクション 19.3 で規定された ACK フレームの拡張である。各パスが独自のパケット番号空間を持つため、異なるパス上で送信されたパケットを確認応答するために使用される。フレームタイプが `0x3f` の場合、PATH_ACK フレームは、現時点までに確認応答されたパケット番号空間で受信された関連付けられた ECN マークを持つ QUIC パケットの合計も含む。

PATH_ACK フレームは図 6 に示すフォーマットで構成される。

```
PATH_ACK Frame {
  Type (i) = 0x3e..0x3f,
  Path Identifier (i),
  Largest Acknowledged (i),
  ACK Delay (i),
  ACK Range Count (i),
  First ACK Range (i),
  ACK Range (..) ...,
  [ECN Counts (..)],
}

                 図 6: PATH_ACK フレームフォーマット
```

[QUIC-TRANSPORT] で規定される ACK フレームと比較して、以下のフィールドが追加されている。

- **Path Identifier**: PATH_ACK フレームによって確認応答される 0-RTT および 1-RTT パケットのパケット番号空間に関連付けられた path ID。

### 4.2. PATH_ABANDON フレーム

PATH_ABANDON フレームは、パスを放棄するようピアに通知する。PATH_ABANDON フレームがパス上で送信された後、当該パスは送信に使用できなくなる。

PATH_ABANDON フレームは図 7 に示すフォーマットで構成される。

```
PATH_ABANDON Frame {
  Type (i) = 0x3e75,
  Path Identifier (i),
  Error Code (i),
}

               図 7: PATH_ABANDON フレームフォーマット
```

PATH_ABANDON フレームは以下のフィールドを含む。

- **Path Identifier**: 放棄予定のパスに関連付けられた path ID。
- **Error Code**: パスを放棄する理由を示す可変長整数。NO_ERROR (0x0) はエラーが発生せずにパスが放棄されていることを示す。他のエラーコードはセクション 4.2.1 にある。

PATH_ABANDON フレームは ack-eliciting である。PATH_ABANDON フレームを含むパケットが失われたとみなされる場合、ピアはそれを繰り返すべきである (SHOULD)。

PATH_ABANDON フレームの使用はセクション 3.4 で規定する。

#### 4.2.1. エラーコード

QUIC トランスポートエラーコードは 62 ビット符号なし整数である ([QUIC-TRANSPORT] のセクション 20.1 参照)。NO_ERROR (0x0) に加えて、以下の QUIC エラーコードが PATH_ABANDON フレームでの使用のために定義される。

- **APPLICATION_ABANDON_PATH (`0x3e`)**: エンドポイントがアプリケーションの要求でパスを放棄している。
- **PATH_RESOURCE_LIMIT_REACHED (`0x3e75`)**: エンドポイントが、それを維持するための十分なリソースを割り当てられないためパスを放棄している。
- **PATH_UNSTABLE_OR_POOR (`0x3e76`)**: エンドポイントが、使用されているインターフェースが不安定であると観察される、または性能が低いとみなされるためパスを放棄している。この状況は、たとえば高速移動中の頻繁なハンドオーバーイベントや弱い無線信号により発生し得る。
- **NO_CID_AVAILABLE_FOR_PATH (`0x3e77`)**: エンドポイントが、当該パス用のコネクション ID の不足のためパスを放棄している。これは、ピアが新しいパスを開始したが対応する path ID 用のコネクション ID を提供していない場合（またはコネクション ID を含むパケットがまだ到着していない場合）に発生し得る。

### 4.3. PATH_STATUS_AVAILABLE および PATH_STATUS_BACKUP フレーム

PATH_STATUS_AVAILABLE フレーム（type=`0x3e77`）は、エンドポイントが指定されたパスが送信に利用可能であることをピアに通知するために使用される。

PATH_STATUS_AVAILABLE フレームは図 8 に示すフォーマットで構成される。

```
PATH_STATUS_AVAILABLE Frame {
  Type (i) = 0x3e77,
  Path Identifier (i),
  Path Status Sequence Number (i),
}

           図 8: PATH_STATUS_AVAILABLE フレームフォーマット
```

PATH_STATUS_BACKUP フレーム（type=`0x3e76`）は、エンドポイントが指定されたパスを送信に使用しない優先順位についてピアに通知するために使用される。

PATH_STATUS_BACKUP フレームは図 9 に示すフォーマットで構成される。

```
PATH_STATUS_BACKUP Frame {
  Type (i) = 0x3e76
  Path Identifier (i),
  Path Status Sequence Number (i),
}

            図 9: PATH_STATUS_BACKUP フレームフォーマット
```

PATH_STATUS_AVAILABLE および PATH_STATUS_BACKUP フレームの両方は、以下のフィールドを含む。

- **Path Identifier**: ステータス更新が対応する path ID。最大 path ID 制限以下のすべての path ID を示すことができる、たとえパスがまだアクティブに使用されていない場合でも。
- **Path Status Sequence Number**: このフレームに割り当てられたパスごとのシーケンス番号を指定する可変長整数。

シーケンス番号空間は 2 つのフレームタイプ間で共通であり、特定の path ID に対して PATH_STATUS_AVAILABLE または PATH_STATUS_BACKUP フレームを送信する際は、単調増加する値を使用しなければならない (MUST)。

フレームは順序が乱れて受信される可能性がある。ピアは、同じ path ID に対して、入力フレームの Path Status シーケンス番号以上の Path Status シーケンス番号を持つ別の PATH_STATUS_BACKUP フレームまたは PATH_STATUS_AVAILABLE フレームを以前に受信していた場合、入力された PATH_STATUS_AVAILABLE または PATH_STATUS_BACKUP フレームを無視しなければならない (MUST)。

単調増加するシーケンス番号の要件はパスごとである。受信側は、異なるパス上で PATH_STATUS_AVAILABLE または PATH_STATUS_BACKUP フレーム用の同じシーケンス番号を受信することがあり得る。したがって、PATH_STATUS_AVAILABLE または PATH_STATUS_BACKUP フレームの受信側は、各 path ID に対して別々にシーケンス番号を使用および比較する必要がある。

PATH_STATUS_BACKUP および PATH_STATUS_AVAILABLE フレームは ack-eliciting である。PATH_STATUS_BACKUP または PATH_STATUS_AVAILABLE フレームを含むパケットが失われたとみなされる場合、ピアは、シーケンス番号で示されるとおり、当該パス用に送信された最後のステータスを含む場合にのみフレームを再送すべきである (SHOULD)。

PATH_STATUS_BACKUP または PATH_STATUS_AVAILABLE フレームは、パス開始の前または間に優先パス使用を示すため、PATH_NEW_CONNECTION_ID フレームまたは PATH_RESPONSE フレームとバンドルしてもよい (MAY)。

### 4.4. PATH_NEW_CONNECTION_ID フレーム

PATH_NEW_CONNECTION_ID フレーム（type=`0x3e78`）は、[QUIC-TRANSPORT] のセクション 19.15 で規定される NEW_CONNECTION_ID フレームの拡張である。これは、特定のパス用の 1-RTT パケットの代替コネクション ID をピアに提供するために使用される。ピアは、当該パス上で異なるコネクション ID を使用して、当該パス上のマイグレーション時のリンカビリティを破ることができる（[QUIC-TRANSPORT] のセクション 9.5 も参照）。

PATH_NEW_CONNECTION_ID フレームは図 10 に示すフォーマットで構成される。

```
PATH_NEW_CONNECTION_ID Frame {
  Type (i) = 0x3e78,
  Path Identifier (i),
  Sequence Number (i),
  Retire Prior To (i),
  Length (8),
  Connection ID (8..160),
  Stateless Reset Token (128),
}

           図 10: PATH_NEW_CONNECTION_ID フレームフォーマット
```

[QUIC-TRANSPORT] のセクション 19.15 で規定される NEW_CONNECTION_ID フレームと比較して、以下のフィールドが追加されている。

- **Path Identifier**: コネクション ID に関連付けられた path ID。これは、提供されたコネクション ID が対応するパス上でのみ使用できることを意味する。

なお、[QUIC-TRANSPORT] のセクション 19.15 の NEW_CONNECTION_ID フレームと異なり、シーケンス番号はパスごとのコンテキストに適用される。これは、異なるパス上の異なるコネクション ID が同じシーケンス番号値を持つ可能性があることを意味する。

Retire Prior To フィールドは、Path Identifier フィールドの path ID を共有するもののうち、どのコネクション ID が退役されるべきかを示す。異なる path ID に関連付けられたコネクション ID は影響を受けない。

NEW_CONNECTION_ID フレームは、初期パス（path ID 0）用のコネクション ID を発行または退役するためにのみ使用できることに注意。

[QUIC-TRANSPORT] のセクション 5.1.2 の最後の段落では、入力 NEW_CONNECTION_ID フレームの Retire Prior To フィールドを検証する方法が規定されている。同じ規則が PATH_NEW_CONNECTION_ID フレームに適用されるが、パスごとに適用される。マルチパス拡張が使用される場合、NEW_CONNECTION_ID フレーム用の規則は path ID 0 にのみ適用される。

### 4.5. PATH_RETIRE_CONNECTION_ID フレーム

PATH_RETIRE_CONNECTION_ID フレーム（type=`0x3e79`）は、[QUIC-TRANSPORT] のセクション 19.16 で規定される RETIRE_CONNECTION_ID フレームの拡張である。これは、エンドポイントがピアによって発行された特定の path ID 用のコネクション ID をもう使用しないことを示すために使用される。初期パス上のハンドシェイク中に使用されたコネクション ID を退役するには、path ID 0 が使用される。PATH_RETIRE_CONNECTION_ID フレームを送信することは、ピアに対し当該パス用の追加コネクション ID を送信するよう要求するものとしても機能する（[QUIC-TRANSPORT] のセクション 5.1 も参照）、ただし path ID で指定されたパスが放棄されていない場合に限る。新しいパス固有のコネクション ID は、PATH_NEW_CONNECTION_ID フレーム（セクション 4.4 参照）を使用してピアに配信できる。

PATH_RETIRE_CONNECTION_ID フレームは図 11 に示すフォーマットで構成される。

```
PATH_RETIRE_CONNECTION_ID Frame {
  Type (i) = 0x3e79,
  Path Identifier (i),
  Sequence Number (i),
}

         図 11: PATH_RETIRE_CONNECTION_ID フレームフォーマット
```

[QUIC-TRANSPORT] のセクション 19.16 で規定される RETIRE_CONNECTION_ID フレームと比較して、以下のフィールドが追加されている。

- **Path Identifier**: 退役するコネクション ID に関連付けられた path ID。

なお、RETIRE_CONNECTION_ID フレームは、初期パス（path ID 0）用のコネクション ID を退役するためにのみ使用できる。

PATH_NEW_CONNECTION_ID フレームがシーケンス番号をパスごとに適用するため、PATH_RETIRE_CONNECTION_ID フレームのシーケンス番号もパスごとである。PATH_RETIRE_CONNECTION_ID フレームは、指定された path ID とシーケンス番号を持つコネクション ID を退役する。

入力 RETIRE_CONNECTION_ID フレームの処理は [QUIC-TRANSPORT] のセクション 19.16 で記述されている。同じ処理が PATH_RETIRE_CONNECTION_ID フレームにパスごとに適用されるが、マルチパス拡張の使用では RETIRE_CONNECTION_ID フレームの処理は path ID 0 にのみ適用される。

### 4.6. MAX_PATH_ID フレーム

MAX_PATH_ID フレーム（type=`0x3e7a`）は、ピアに対し使用が許可される最大の path ID を通知する。

MAX_PATH_ID フレームは図 12 に示すフォーマットで構成される。

```
MAX_PATH_ID Frame {
  Type (i) = 0x3e7a,
  Maximum Path Identifier (i),
}

                図 12: MAX_PATH_ID フレームフォーマット
```

MAX_PATH_ID フレームは以下のフィールドを含む。

- **Maximum Path Identifier**: 送信側エンドポイントが受け入れる意思のある最大の path ID。この値は、ノンス計算上の制約（セクション 2.4 参照）により path ID の最大許容値である 2^32-1 を超えてはならない (MUST NOT)。Maximum Path Identifier の値は、`initial_max_path_id` トランスポートパラメータで通知された値より低くてはならない (MUST NOT)。

無効な Maximum Path Identifier 値の受信は、タイプ PROTOCOL_VIOLATION のコネクションエラーとして扱わなければならない (MUST)。

損失または順序が乱れた配信により、エンドポイントは以前に受信したものより小さい Maximum Path Identifier 値を持つ MAX_PATH_ID フレームを受信する可能性がある。パス制限を増加させない MAX_PATH_ID フレームは無視しなければならない (MUST)。

MAX_PATH_ID フレームは ack-eliciting であり、損失した場合、その間に最新の MAX_PATH_ID フレームが送信されていないなら再送すべきである (SHOULD)。

### 4.7. PATHS_BLOCKED および PATH_CIDS_BLOCKED フレーム

送信者は、パスを開きたいが、ピアによって設定された最大 path ID 制限のために開けない場合、PATHS_BLOCKED フレーム（type=`0x3e7b`）を送信できる。

送信者は、有効な path ID でパスを開くか、または確立されたパス上でコネクション ID を変更したいが、対応する path ID 用の利用可能な未使用コネクション ID がないために開けない場合、PATH_CIDS_BLOCKED フレーム（type=`0x3e7c`）を送信できる。

なお、PATHS_BLOCKED および PATH_CIDS_BLOCKED フレームは情報提供である。PATHS_BLOCKED または PATH_CIDS_BLOCKED フレームの送信は、新しい Maximum Path Identifier 値を持つ MAX_PATH_ID フレームの送信などのピアからの特定のアクションを意味するわけではないが、最大 path ID 制限または未使用コネクション ID の不足によりパスの作成または使用が妨げられたことをピアに通知する。PATHS_BLOCKED/PATH_CIDS_BLOCKED フレームの受信が確認応答されたが、ピアによって何らアクションが取られない場合、これはピアによる意図的な決定である可能性が高く、PATHS_BLOCKED/PATH_CIDS_BLOCKED フレームを繰り返してもそれを変えることはない。

PATHS_BLOCKED フレームは図 13 に示すフォーマットで構成される。

```
PATHS_BLOCKED Frame {
  Type (i) = 0x3e7b,
  Maximum Path Identifier (i),
}

               図 13: PATHS_BLOCKED フレームフォーマット
```

PATHS_BLOCKED フレームは以下のフィールドを含む。

- **Maximum Path Identifier**: フレームが送信された時点で許可された最大 path ID を示す可変長整数。受信した値が現在許可されている最大値より低い場合、このフレームは無視できる。

PATH_CIDS_BLOCKED フレームは図 14 に示すフォーマットで構成される。

```
PATH_CIDS_BLOCKED Frame {
  Type (i) = 0x3e7c,
  Path Identifier (i),
  Next Sequence Number (i),
}

             図 14: PATH_CIDS_BLOCKED フレームフォーマット
```

PATH_CIDS_BLOCKED フレームは以下のフィールドを含む。

- **Path Identifier**: 未使用コネクション ID が利用できないパスの識別子。
- **Next Sequence Number**: ピアによって当該パス用に発行されることが期待される次のシーケンス番号。

ローカルの最大値より高い Maximum Path Identifier または Path Identifier 値の受信は、タイプ PROTOCOL_VIOLATION のコネクションエラーとして扱わなければならない (MUST)。

当該パス用に次に発行されることが期待されるコネクション ID のシーケンス番号より高い Next Sequence Number の値の受信は、タイプ PROTOCOL_VIOLATION のコネクションエラーとして扱わなければならない (MUST)。

PATHS_BLOCKED および PATH_CIDS_BLOCKED フレームは ack-eliciting であり、損失が検出された時点でパスがまだブロックされている場合、再送してもよい (MAY)。

## 5. 実装上の考慮事項 (Implementation Considerations)

このセクションは実装者に対する情報提供のガイダンスを提供する。

### 5.1. コネクション ID の変更、マイグレーション、および NAT リバインディング

マルチパス拡張では、各パスは個別のパケット番号空間を使用する。これは [QUIC-TRANSPORT] からの大きな違いである。[QUIC-TRANSPORT] では 3 つの番号空間（Initial、Handshake、Application データパケット）のみが定義されている。

特定のパスに対して、コネクション ID のローテーション、NAT リバインディング、または [QUIC-TRANSPORT] で規定されるクライアント開始のマイグレーションが、単一パスのように発生する可能性がある。これらのイベントは path ID を変更せず、パスに関連付けられたパケット番号空間に影響を与えない。

[QUIC-TRANSPORT] で規定される単一パスのマイグレーションを行うよりも、新しいパスを作成して後で古いパスを放棄するなどのマルチパスメカニズムを使用することが一般に好ましい。これにより、よりスムーズなハンドオーバーが可能になり、サーバ側でより制御されたマイグレーション処理が可能になる。ただし、NAT リバインディングの場合、またはサーバがハンドシェイク中に「優先アドレス」へのマイグレーションを要求する場合、単一パスのマイグレーションは避けられない。

[QUIC-TRANSPORT] のセクション 9.3 では、最近見たアドレスである場合、エンドポイントがピアアドレスの検証をスキップすることを許可している。ただし、マルチパス拡張が使用され、エンドポイントが異なるパス間の切り替えにつながる可能性のある複数のアドレスを持つ場合、代わりに複数の開いたパスを維持すべきである。

4-tuple の変更を観察するサーバはパス検証を行う（[QUIC-TRANSPORT] のセクション 9 参照）。パス検証プロセスが成功すると、エンドポイントは [QUIC-TRANSPORT] のセクション 9.4 に従ってパスの輻輳制御器とラウンドトリップ時間推定器をリセットする。

### 5.2. 同じ 4-tuple 上で複数のパスを使用する

同じ 4-tuple を参照するパスを作成することが可能である。たとえば、エンドポイントは異なる Differentiated Service [RFC2475] マーキングを使用するパスを作成したい場合がある。これは、ストリームをパスにマッチさせるスケジューリングアルゴリズムと組み合わせて行うことができ、たとえば低優先度ストリームのデータフレームが低優先度パス上で送信される。これらのパスは異なる path ID を使用するため、アプリケーションのニーズに合わせて独立に管理できる。（アプリケーションは、クライアントとサーバがパス上で differentiated services をどのように使用するかを管理する必要がある。これは本文書では規定しない。）

異なる 4-tuple で作成されたパスが、パスマイグレーションの結果として同じ 4-tuple を使用するようになる場合がある。すべてのパスが同じ送信元および宛先ポートを使用する以下の例を考える。

- クライアントがアドレス 192.0.2.1 からサーバアドレス 198.51.100.1 へのパス 1 を開始
- クライアントがアドレス 192.0.2.2 からサーバアドレス 198.51.100.1 へのパス 2 を開始
- 両方のパスがしばらく使用される
- サーバがアドレス 198.51.100.1 からクライアントアドレス 192.0.2.1 へ、path ID 2 を示すコネクション ID でパケットを送信
- クライアントがパケットを受信し、パスマイグレーションを認識し、パス 2 の送信元アドレスを 192.0.2.1 に更新

（簡潔さのため、この例では IPv4 アドレスを使用しているが、IPv6 アドレスでも同様に動作する。）

このような異なるパス上での同じ 4-tuple の意図しない使用はまれであるべきである。それらが発生した場合、2 つのパスは冗長であり、エンドポイントはそれらの 1 つをクローズしたい場合がある。

### 5.3. 輻輳制御

QUIC マルチパス拡張が使用される場合、送信者は [QUIC-TRANSPORT] のセクション 9.4 で要求されるとおり、パスごとの輻輳ステータスを管理する。ただし、[QUIC-TRANSPORT] では 1 つのアクティブパスのみが想定されており、要件はパスマイグレーション時に輻輳制御ステータスをリセットすることである。マルチパス拡張では、複数のパスを同時に使用できるため、各パスに対して個別の輻輳制御状態が維持される。これは、送信者は当該パスの輻輳制御が示すよりも多くのデータを特定のパス上で送信することは許されないことを意味する。

マルチパス QUIC コネクションが 2 つ以上のパスを使用する場合、これらのパスが完全に分離されている保証はない。2 つ（またはそれ以上）のパスが同じボトルネックを共有する場合、標準の輻輳制御方式を使用すると、マルチパスコネクションが競合する単一パスコネクションより多くの帯域幅を獲得する不公平な帯域幅分配につながる可能性がある。Multipath TCP は、この問題を解決するため [RFC6356] で規定されるリンク増加アルゴリズム (LIA) 輻輳制御方式を使用する。この方式は Multipath QUIC に直ちに適用できる。Multipath TCP 用には [OLIA] のような他の結合輻輳制御方式も提案されている。Multipath QUIC 専用の輻輳制御アルゴリズムの設計者は BCP 133 に従うことが推奨される（[RFC9743] のセクション 7.10 参照）。

[QUIC-TRANSPORT] のセクション 5.1.2 では、エンドポイントは使用するコネクション ID をコネクション中いつでも別の利用可能なものに変更できることを示している。したがって、アドレスを変更しないコネクション ID の変更はパス変更を示さず、エンドポイントは同じ輻輳制御および RTT 測定状態を維持できる。

### 5.4. パス RTT の計算

PATH_ACK フレームは確認応答されたパケットがどのパス上で送信されたかを示すが、それらはいずれの開いたパスを通じて受信されてもよい。連続する確認応答が異なるパスで受信される場合、測定された RTT サンプルは大きく変動する可能性があり、これは、たとえば輻輳制御アルゴリズムに依存して低い性能をもたらす可能性がある。

[QUIC-RECOVERY] で定義される輻輳制御状態は path ID ごとに保持される。ただし、確認応答がどのパスで送信されるかに依存して、パスの実際の RTT は計算できないか、使用すべき正しい値ではない可能性がある。

パスの実際の RTT を使用する代わりに、2 つの片道遅延の合計を考慮することが推奨される。すなわち、パケット送信パスの遅延と、確認応答に選択された戻りパスの遅延である。異なるパスが異なる特性を持つ場合、遅延は大きく変動する可能性がある。たとえば、各方向 50ms のレイテンシを持つ地上パスと、各方向 300ms のレイテンシを持つ静止衛星パスの両方を使用するマルチパス送信を考える。2 つの片道遅延の合計は、表 1 に示すように、パケット送信および確認応答送信に使用されるパスの組み合わせに依存する。

| ACK パス \ データパス | 地上 | 衛星 |
|---|---|---|
| 地上 | 100ms | 350ms |
| 衛星 | 350ms | 600ms |

**表 1: 複数パスを使用した ACK 遅延の例**

計算された値はネットワークパスの状態と確認応答送信側のスケジューリング判断の両方を反映する。PATH_ACK が地上リンク上で送信されると仮定すると（この判断が最良の応答時間を提供するため）、衛星パスの計算 RTT 値は約 350ms となる。これは PATH_ACK が衛星チャネル経由で来た場合に測定される 600ms より低いが、たとえば PTO タイムアウトを計算するための正しい値である。すなわち、PATH_ACK が 350ms 以上後に受信されない場合、パケットまたはその PATH_ACK のいずれかがおそらく失われた。

最も簡単な実装は、PATH_ACK フレームが受信されるパスにかかわらず、新しいパケット確認応答を受信したときに測定された遅延を使用して [QUIC-RECOVERY] のセクション 5.3 に従って `smoothed_rtt` および `rttvar` を計算することである。確認応答が一貫して 1 つのパス上で送信される限り、このアプローチは良い結果を提供する。確認応答送信側がいつでも送信優先順位を再考する場合、これは確認応答を送信するために使用されるパスも変更する可能性がある。ただし、これは単一パス上のルート変更とそれほど変わらない。RTT、RTT 分散、PTO 推定値は新しい状況を反映するように急速に収束する。1 つの例外がある。最小 RTT であり、これは単一パス上でルート変更が発生する場合にもよく知られた課題である。ただし、確認応答受信側は、最小 RTT を生成した PATH_ACK を受信したパスを記憶し、その確認応答パスが変更または放棄された場合に最小 RTT 計算を再開できる。確認応答が一貫して 1 つのパス上で送信されない場合、確認応答受信側はどのパス上で確認応答が受信されるかを監視し、データが送信されたのと同じパス上で受信された確認応答のサンプルのみを使用できる（あれば）。

さらに、遅延推定に依存する輻輳制御関数は、異なる遅延を持つ複数のパス上で確認応答が送信される場合を明示的に考慮する必要がある。

### 5.5. パケットスケジューリング

データを含むパケットの送信は、アプリケーションからのデータ到着および輻輳制御によって制限される。一般に、転送中のバイト数を増やす QUIC パケットは、選択されたパスの輻輳ウィンドウが許可する場合にのみ送信できる。

ほとんどのフレーム（PATH_CHALLENGE および PATH_RESPONSE は注目すべき例外）は、いずれの開いたパス上でも送信および受信できる。したがって、輻輳ウィンドウが開いているパスの中から、次のパケットを送信するためにどのパスを使用するかを決定するパケットスケジューラが必要である。同じストリームに属するデータフレームを送信するために複数のパスが使用される場合、順序通り配信のため、データ配信は使用されるすべてのパスの最大遅延を経験する。スケジューリングは、アプリケーションおよび実装の優先順位に基づくローカル決定である。

これは、エンドポイントが確認応答されたパケットを運んだものとは異なるパス上で PATH_ACK フレームを送受信する可能性があることを意味する。セクション 5.4 で述べたとおり、標準アルゴリズムを使用して計算された RTT 推定値は、パスの特性と PATH_ACK フレームのスケジューリングアルゴリズムの両方を反映する。PATH_ACK フレームのスケジューリング戦略が安定している場合、推定値はより速く収束する。実装は、たとえば、確認応答されたパケットが受信されたパス上で単に PATH_ACK フレームを送信する、または代替として最短パスで送信するなど、異なる戦略を選択できる。最短パスは、より短い制御ループとおそらくより良い性能をもたらす。

PATH_ACK フレームのみを運ぶパケットは輻輳制御されない（[QUIC-RECOVERY] のセクション 7 参照）ため、送信者は、特に当該パスで容量が不明な場合（たとえば、当該パスがデータフレームの送信に使用されていない場合）、これらのパケットによって誘発される負荷を慎重に考慮すべきである。

### 5.6. 再送

複数のパスの同時使用により、損失に対処するための異なる再送戦略が可能となる。たとえば、(a) 同じパス上で失われたフレームを再送する、(b) 異なるまたは専用のパス上で失われたフレームを再送する、(c) 複数のパス上で失われたフレームを複製する（ネットワークオーバーヘッドのため一般用途には推奨されない）。本文書は特定の戦略を排除するものではないが、より詳細な仕様は範囲外である。

[QUIC-TRANSPORT] のセクション 2.2 で述べたとおり、データが再送される場合、STREAM フレーム境界が保持されることは期待されない。特に、より小さい MTU 制限を持つ異なるパス上で STREAM フレームを再送する必要がある場合、より小さい STREAM フレームを代わりに送信する必要がある場合がある。

### 5.7. PTO の満了

実装は、各個別パスでパケット損失を検出するために [QUIC-RECOVERY] で規定されるメカニズムに従うべきである。特殊なケースは PTO タイマーが満了したときに発生する。[QUIC-RECOVERY] によると、パケット送信者がこのパス用の新しい確認応答を受信するか、またはパス自体が最終的に壊れたと宣言されるまで、パケットは失われたと宣言されない。この慎重なプロセスは偽の再送のリスクを最小化するが、これらの「失われたパケット」に含まれるフレームの配信遅延が大きくなる可能性がある。

エンドポイントはマルチパス拡張を活用し、これらのパス上の輻輳制御ウィンドウが許可する場合、他の利用可能なパス上で遅延したパケットの内容を再送できる。

### 5.8. 異なる PMTU サイズを持つパス

実装は、複数のパス間で異なる PMTU サイズの取り扱いに注意すべきである。[QUIC-TRANSPORT] のセクション 14.3 で規定されるとおり、DPLPMTUD Maximum Packet Size (MPS) はローカルおよびリモート IP アドレスの各組み合わせに対して維持される。なお、マルチパス拡張では、複数のパスが同じ 4-tuple を使用できるが、他の要因により異なる MPS を持つ可能性がある（セクション 5.2 参照）。

PMTU が類似している場合の 1 つの簡単なオプションは、すべてのパスの最小 PMTU を各パスに適用することであり、これは再送処理の簡素化にも役立ち得る。

### 5.9. アイドルタイムアウトとキープアライブ

[QUIC-TRANSPORT] はコネクションをクローズするためのアイドルタイムアウトを定義しており、これは、いずれのパス上でもアイドルタイムアウトの期間中パケットが受信されない場合、マルチパス使用に適用される。

本文書はパスごとのアイドルタイムアウトを規定しない。エンドポイントは、パスがアクティブに使用されているかどうかにかかわらず、いつでもパスをクローズすることを判断できる。たとえば、エンドポイントはいずれにせよ別のフレームを送信するまで、初期 PATH_ABANDON フレームの送信を待ってもよい。ただし、初期 PATH_ABANDON フレームの受信側は直ちに応答する必要があることに注意（セクション 3.4 参照）。

パスがしばらくアクティブに使用されない場合、たとえばミドルボックスのタイムアウトのために、もう使用できなくなる可能性がある。そのようなパス断絶を避けるため、エンドポイントは PING フレーム（[QUIC-TRANSPORT] のセクション 19.2）を含むパケットなど、ack-eliciting なパケットを当該パス上で送信して維持できる。ただし、本仕様は、特に他のアクティブに使用されているパスがある場合、不要なオーバーヘッドを生成する可能性があるため、キープアライブの送信を推奨しない。

[QUIC-TRANSPORT] のセクション 5.3 はオプションのキープアライブプロセスを定義している。このプロセスは、アプリケーションのニーズに応じて各パスに別々に適用できる。一部のアプリケーションは、アクティブに使用されていないパスを維持しないこと、追加の 1 つだけのパスを維持すること、または冗長性のために複数のパスを維持することを判断する場合がある。[QUIC-TRANSPORT] のセクション 10.1.2 で論じられたとおり、キープアライブ間隔はパス上のミドルボックスのタイムアウトを組み込む必要がある。

パスがしばらくアクティブに使用されておらず、キープアライブが送信されていない場合、現在使用可能な他のパスがまだあるなら、エンドポイントはアクティブ使用に切り替える前にプローブできる。

## 7. セキュリティに関する考慮事項 (Security Considerations)

マルチパス拡張は [QUIC-TRANSPORT] および [QUIC-TLS] のすべてのセキュリティ特性を保持するが、以下に関して追加の考慮が必要である。

- パスごとのコネクション ID および複数の同時パスコンテキストに対する潜在的な追加リソース使用量
- 複数のパスが同時に使用される場合、サービス拒否攻撃に対する潜在的に増加した増幅リスク
- 複数のパケット番号空間の使用によるノンス計算の変更

### 7.1. パスごとのリソースのメモリ割り当て

`initial_max_path_id` または MAX_PATH_ID フレームの最大 path ID 制限は、エンドポイントが維持する意思のあるパス数を制限し、それによって関連付けられたパスリソースも制限する。さらに、エンドポイントが新しいパスを開く前に同じ path ID 用のコネクション ID が両エンドポイントによって発行される必要があるため、各エンドポイントは限られた数のパス用のコネクション ID のみを発行することによってパスごとのリソース使用量を制御することもできる。ただし、`initial_max_path_id` または MAX_PATH_ID フレームの最大 path ID 制限を使用することが好ましい。

リソース枯渇攻撃で悪用される可能性のある不要なリソース使用を回避するため、エンドポイントは、パケット番号処理などの追加のパスリソースを、パス検証が正常に完了した後にのみ割り当てるべきである (SHOULD)。

### 7.2. 複数パスを伴うサービス拒否攻撃 (Denial of Service with Multiple Paths)

[QUIC-TRANSPORT] のセクション 8.2 で規定されるマイグレーション用のパス検証は、本拡張のパス開始用に変更されずに使用される。さらに、マルチパス拡張は複数のパスの作成を許可する。これは、[QUIC-TRANSPORT] のセクション 21.5.4 で概説された送信元アドレスのなりすましに関するセキュリティ上の考慮事項に加えて、複数パスの同時オープンまたはマイグレーションを通じた増幅型 DoS 攻撃のリスクがあることを意味する。たとえば、攻撃者は複数パスで使用される 4-tuple を設定またはなりすますことで、サーバが送信するパケットが共通のネットワークパスを通過してターゲットを圧倒しようとすることができる。

[QUIC-TRANSPORT] は 1 つのパスの使用のみを許可し、同時パス検証試行の数は発行されたコネクション ID の数によって制限される。本拡張は、理論上同時にすべて移行される可能性のある複数の開いたパスを許可するが、これは依然として各パス用に発行されたコネクション ID の数によって制限される。さらに、複数のパスは同時に初期化される可能性があり、これは許可される最大パス数によって制限される。

したがって、エンドポイントはパスの最大数を小さく保つことが推奨され、たとえばそれらをペースアウトする、または一定期間内のパス開始試行の数を制限するなど、同時パス検証プロセスの数を制限するための追加の措置を考慮する場合がある。

ただし、マルチパスの使用は [QUIC-TRANSPORT] のセクション 8 で規定されるアンチアンプリフィケーション制限を変更しない。攻撃者は攻撃に使用される各パス上で個別のチャレンジを送信する必要があり、増幅制限はサーバが当該パス上で送信するデータ量に適用される。攻撃者はマルチパスオプションをネゴシエートせずに、多数の QUIC コネクションを開いてそれぞれで攻撃を行うことで同じ効果を得ることができる。

### 7.3. 暗号化ハンドシェイクと AEAD ノンス

本文書で規定するマルチパス拡張は、ハンドシェイクが成功し両エンドポイントがこの拡張のサポートを示した後にのみ有効になる。本拡張で定義されるすべての新しいフレームは 1-RTT パケットでのみ使用される。

本拡張ではハンドシェイクが変更されないため、暗号鍵交換やピア認証など [QUIC-TLS] で規定されるトランスポートセキュリティメカニズムは変更されない。したがって、[QUIC-TLS] のセキュリティ上の考慮事項は変更されずに適用される。

[QUIC-TLS] の付録 B で論じられている制限は、各パス別ではなく、すべてのパス上で送信されたパケットの合計数に適用される。

本仕様は、AEAD ノンス計算に path ID を計算の一部として含めることによって変更する（セクション 2.4 参照）。一意のノンスを保証するため、path ID は 32 ビットに制限され、同じコネクションの別のパスに再利用できない。
